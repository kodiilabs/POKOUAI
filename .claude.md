# PokouAI — Claude Configuration

## Project Overview
**PokouAI** is an offline-first, multilingual AI crop disease advisor for West African smallholder farmers. Built for the Gemma 4 Good Hackathon (deadline: May 18, 2026).

- **Primary crop**: Cocoa (Ivory Coast)
- **Primary target**: Farmers on entry-level Android/iOS with 2G or no internet
- **Core constraint**: 100% offline inference via llama.cpp

---

## Tech Stack & Conventions

### Backend / ML Pipeline
- **Language**: Python 3.11 (via pyenv)
- **Package manager**: pip (within venv)
- **ML framework**: PyTorch, Unsloth, transformers
- **Model**: Gemma 4 **E2B** (on-device primary, ~1.5 GB Q4_K_M), Gemma 4 E4B (optional premium for ≥4 GB RAM phones), Gemma 4 27B MoE (cloud fallback)
- **Licensing**: Gemma 4 weights load from `google/gemma-4-e2b-it` / `google/gemma-4-e4b-it` directly (no click-through gating). Notebook falls back to Unsloth 4-bit repos if the direct load errors.
- **Quantization**: GGUF Q4_K_M (llama.cpp format)
- **Training platform**: Kaggle (free GPU T4 x2)
- **Fine-tuning method**: QLoRA (r=16, alpha=32)

**Key files**:
- `prepare_training_data.py` — data pipeline
- `augment_images.py` — image augmentation for training robustness
- `cocoa_diseases.json` — disease knowledge base (symptoms, treatments, prevention)
- Kaggle notebooks: `03_finetune_cocoa_unsloth.ipynb`

### Mobile App
- **Framework**: React Native + Expo (iOS + Android unified codebase)
- **Language**: TypeScript
- **Package manager**: npm (via nvm, Node 20)
- **Key packages**: 
  - `expo-camera`, `expo-image-picker` (image capture)
  - `react-native-llama.cpp` (on-device inference)
  - `@react-native-async-storage/async-storage` (preferences + offline state)
  - `expo-sqlite` (local farm log database)
  - `i18next` + `react-i18next` (multilingual UI)
  - `react-native-whisper` (voice input, planned v1.1)
- **Internationalization**: Four language files (`fr.json`, `dyu.json`, `bci.json`, `en.json`)
- **App structure**:
  ```

      OnboardingScreen.tsx
      HomeScreen.tsx

      ResultScreen.tsx
      FarmLogScreen.tsx

    services/

      cocoa_diseases.json
  ```

## Critical Constraints

### Offline-First Design
- **Rule**: Every feature must work with zero internet. Cloud is enhancement only.
- **Test**: Always verify app launches and performs diagnosis in airplane mode.
- **Storage**: GGUF model bundled in app (~2.5–3.0 GB) or lazy-downloaded on first launch.

### Multilingual Support
- **Four languages required for v1**: French, Dioula, Baoulé, English
- **Dioula & Baoulé**: Low-resource languages — use native speaker review
- **Response format**: All disease responses must include (1) disease name, (2) treatment steps, (3) prevention advice in all 4 languages
- **UI strings**: Use i18n keys, never hardcode text

### Hardware Constraints
- **Target device**: 2–3 GB RAM entry-level Android — this is what determines the default model choice
- **Primary model**: Gemma 4 **E2B** Q4_K_M (~1.5 GB, runs in ~2 GB RAM). Chosen because E4B Q4_K_M (~2.8 GB) would leave no headroom for the OS and camera pipeline on the target device.
- **Premium model**: Gemma 4 E4B Q4_K_M for phones with ≥4 GB RAM (optional download in Settings)
- **Inference time goal**: <15 seconds per image on emulator (faster on real device)
- **Image size**: Max 1024px (limit memory usage + transmission)

## Code Style & Standards

### Python
- PEP 8 compliance
- Type hints on all functions
- Docstrings for all public functions / modules
- No hardcoded paths — use `pathlib.Path` and environment variables

### TypeScript / React Native
- ESLint + Prettier (check `.eslintrc.json`, `.prettierrc`)
- Strict null checking (`strict: true` in `tsconfig.json`)
- No `any` types without explicit comment
- Component naming: PascalCase
- Variable/function naming: camelCase
- Export screens/services as named exports

### JSON (disease knowledge base, i18n files)
- Organized by disease
- Flat key structure for i18n (e.g., `disease.black_pod.name`, `disease.black_pod.treatment_steps`)
- Verify all 4 languages present before commit

## When Generating Code

1. **Always ask about constraints** if unclear:
   - Device RAM assumptions?
   - Internet connectivity assumptions?
   - Language priority?

2. **Multilingual first**: Never hardcode English. Use i18n keys from day one.

3. **Offline-first**: Assume no internet. Cloud features are bonus.

4. **Performance matters**: Document inference time, model size, RAM footprint.

5. **Schema consistency**: Disease responses must match the structure in `cocoa_diseases.json`.

6. **Test the path**: For mobile features, confirm they work in emulator before suggesting.
